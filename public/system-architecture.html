⚡ V4 後續缺口分析

以下是目前流程中缺少或未完成的 V4 功能模組

1. DAG 依賴執行引擎（dag_executor.py 存在但未串接）
V4 設計中任務之間有依賴關係（A 完成才能開始 B），dag_executor.py 已寫好但 pipeline_engine_v2 沒有呼叫它。需要：將 DAG 邏輯整合進派發流程，讓 phase_unlock.py 和 dag_executor.py 形成閉環。
2. Phase 3 自動化（phase3_runner.py 孤立）
V4 分三個 Phase，Phase 1-2 的自動化已完成，但 Phase 3（進階功能）的 runner 沒有被任何 cron 或工作流觸發。需要：定義 Phase 3 任務清單 + 觸發條件 + 掛進 pipeline。
3. 主動任務管理（proactive_task_management.py 孤立）
Travis 應該主動發現系統問題並建任務卡（intention_engine_v1.py 已有原型），但 proactive_task_management.py 沒有被整合。需要：與 intention_engine 合併，掛進 heartbeat 或每日例行。
4. Agent 間協作鏈不完整
目前 Agent 只能透過 Travis 中轉通訊（notify_agent.sh 很少使用）。V4 設計中 Blake 完成開發後應直接通知 Griffin 審計，但實際上沒有自動觸發。需要：task_complete.sh 完成後根據任務類型自動觸發下游 Agent。
5. 成本/Token 預算閉環
token_threshold_monitor.py 和 budget_monitor.py 只做監控告警，沒有自動限流。當 token 用量超標時應自動降頻或暫停低優先任務。需要：budget_monitor → pipeline_engine_v2 的速率限制聯動。
6. Hooks 系統未啟用
hook_trigger.sh 存在但沒有任何 session 啟動/結束時自動觸發。AGENTS.md 描述的 session_start/session_end hooks 是設計但未實現。需要：在 OpenClaw config 中設定 session lifecycle hooks。
7. Zimbra 簽核自動化斷裂
check_zimbra_approval.py 存在但其 cron wrapper（714_zimbra_approval.sh）已刪除。需要：重新建立 cron 條目，工作日 09:00-18:00 每 30 分鐘檢查簽核郵件。
