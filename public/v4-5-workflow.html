已知問題

漏洞（Task #3748 已修補）：subagent 直接用 supabase_sql.sh INSERT INTO board_tasks 建卡，繞過 create_task.sh，導致 dispatch_prompt 缺失。
修補方式：Supabase trigger（board_tasks INSERT 時若 dispatch_prompt IS NULL 寫入 task_warnings 表）+ dispatch_prompt 模板末尾禁止直接 SQL 建卡。

注意事項

所有建卡必須走 ~/clawd/scripts/create_task.sh，不得直接 SQL INSERT。dispatch_prompt、acceptance_criteria 為必填欄位。

§3 L2 任務形成層（路由決策）

L2 是 Travis 主控的判斷層。任務進入 board_tasks 後，由 Travis（或 auto_dispatch.sh）判斷走哪條路。

路由判斷標準

條件路由理由
子任務數 1–2 個、單一 agent 即可完成、<30 分鐘L3A sessions_spawn低成本、快速
子任務 ≥3 個、需要並行、需要獨立 git branchL3B ClawTeam隔離執行、真正並行
影響生產環境、金額 >$X、架構決策通知 William需人工確認

分解邏輯（尚未實作，需開發）

目標：智能任務分解器
Travis 收到任務後，分析 dispatch_prompt 內容，自動判斷：
  1. 需要幾個子任務（NLP 分析或規則判斷）
  2. 子任務間是否有依賴關係（DAG）
  3. 路由到 L3A 或 L3B
⚠️ 現況：Travis 仍靠人工判斷。自動分解器尚未實作，為 V4.5 後續開發項目。

§4 L3A 輕量執行（sessions_spawn）

現有機制，已在生產環境運作。每次 spawn 在 OpenClaw 進程內建立新 session。

標準派發流程

# Travis 派發（主 session 內呼叫）
sessions_spawn(
  task="完整 dispatch_prompt 內容",
  label="task-{id}-{name}",
  mode="run",
  model="openai-codex/gpt-5.4"   # 或 grok / kimi
)

# Blake/Rex 完成後必須呼叫
bash ~/clawd/scripts/task_complete.sh {TASK_ID} "完成摘要"
⚠️ 同時上限：最多 2 個 subagent 同時執行（AGENTS.md §16 Token 節流規則）。超過會排隊等待。

結果回傳到 L4

task_complete.sh 會自動：更新 board_tasks status→已完成、寫入 result 欄位、發 Telegram 通知、觸發下一輪 auto_dispatch。

§5 L3B ClawTeam Swarm 執行

重要：基於原始碼審查的關鍵差異
ClawTeam workers 不是 OpenClaw session，而是在 tmux 開新 window 啟動獨立 openclaw CLI 進程。
• Worker 不繼承主 session 的 AGENTS.md、SOUL.md、MEMORY.md
• Worker 透過 ClawTeam 的 prompt injection 取得身份和任務
• 所有 agent 間通訊走 clawteam inbox，不走 Telegram

5.1 架構圖（基於 tmux_backend.py）

# 實際執行結構（原碼確認）
tmux session: clawteam-{team_name}
├── window 0: leader-agent   ← openclaw CLI 進程
├── window 1: worker-blake   ← openclaw CLI 進程
│   env: CLAWTEAM_AGENT_ID, CLAWTEAM_AGENT_NAME, CLAWTEAM_TEAM_NAME
│   cwd: ~/clawd (git worktree branch: clawteam/{team}/blake)
├── window 2: worker-rex     ← openclaw CLI 進程
└── window 3: worker-oscar   ← openclaw CLI 進程

5.2 Worker Prompt 注入（基於 prompt.py）

每個 worker 啟動時自動注入的 prompt（spawn/prompt.py build_agent_prompt()）：

## Identity
- Name: blake
- Type: worker
- Team: my-team
- Leader: travis-leader

## Workspace
- Working directory: ~/clawd
- Branch: clawteam/my-team/blake   ← 獨立 git worktree

## Task
{task 內容}

## Coordination Protocol（自動注入）
- clawteam task list my-team --owner blake
- clawteam task update my-team <id> --status in_progress
- clawteam inbox send my-team travis-leader "完成了 X"
- clawteam cost report my-team --input-tokens N --output-tokens N
⚠️ AGENTS.md 不會自動繼承
Worker 只拿到 ClawTeam 注入的 prompt，不會讀取 AGENTS.md/SOUL.md。
解法：在 TOML 模板的 system_prompt 欄位加入我們的核心規則（至少是禁止直接 SQL 建卡、task_complete.sh 必須呼叫等關鍵規則）。

5.3 Git Worktree 隔離（基於 workspace/git.py）

# ClawTeam 執行的 git 指令（原碼確認）
git worktree add -b clawteam/my-team/blake \
  ~/clawd/.clawteam/my-team/blake HEAD

# 前提：~/clawd 必須是 git repo ← 已確認 ✅
# 注意：每個 worker 的 worktree branch 不能有未 commit 的衝突
worktree 路徑預設是 ~/clawd/.clawteam/{team}/{agent}。Worker 完成後需 merge 或 cherry-pick 回主 branch，ClawTeam 目前不自動合併。

# 建立 travis-team.toml（見 §5.5）
4整合 EMERGENCY_STOP 到 ClawTeam

  --team test-team \

§8 已知限制與風險

問題影響解法優先級
Workers 不繼承 AGENTS.md Worker 可能不遵循我們的行為規則 在 TOML system_prompt 注入關鍵規則 P0 必修
ClawTeam 不受 EMERGENCY_STOP 控制 /stop 無法停止執行中的 swarm 修改 emergency_stop.sh 加入 tmux kill P0 必修
git worktree 需要 clean 狀態 ~/clawd 有未 commit 檔案會導致 worktree 建立失敗 派發前確認 git status 乾淨 P1
Worker 結果不自動 merge 回主 branch Worker 的程式碼改動留在獨立 branch Leader 任務 prompt 需包含 merge 步驟 P1
Token 成本翻倍 3 個 workers = 3 倍 token 用量 只有真正需要並行的任務才走 ClawTeam P2
tmux 環境依賴 無 tmux 則完全無法使用 tmux 已安裝，確認版本即可 已確認
